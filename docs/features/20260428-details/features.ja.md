# 機能要件定義: クエリ実行結果詳細とタスク単位ドリルダウン

日付: 2026-04-28
ステータス: Draft

## 1. 背景

- 現在の実行履歴画面は、クエリ実行を 1 件のまとまりとして見せるのではなく、生のタスク実行レコードをそのまま表示している。
- ユーザーは、オーケストレーション完了後にクエリの最終実行結果を確認できない。
- クエリが複数の DAG タスクに分解された場合でも、管理コンソール上で各タスクの実行結果を確認できない。
- 現在の DAG 画面とライブトレース画面は部分的な実行状況しか見せておらず、同じ run の最終結果、タスク単位の詳細、DAG を 1 画面で確認できる詳細画面が存在しない。

## 2. 目的

- 各クエリ実行を 1 件のユーザー向け記録として扱う、run 単位の履歴体験を提供する。
- 各 run に対して専用のクエリ実行結果詳細画面を提供する。
- ユーザーが、最終的なクエリ実行結果、実行したタスク一覧、各タスクの各種パラメータと実行結果、当該 run の DAG を確認できるようにする。

## 3. 用語定義

- Query Run: 共通の `run_id` で識別される 1 回のオーケストレーション実行。
- Final Query Result: Query Run に対する、ユーザー向けの正規の最終結果。システムは、最終レポートを意図した終端タスクの結果を優先して採用する。明示的にそのタスクを特定できない場合でも、終端 DAG タスクまたは終端の失敗状態から決定的に最終結果を導出しなければならない。
- Display Task: Planner の DAG `task_id` をキーとする、ユーザー向けの 1 件の表示タスク。バックエンドが同じ DAG タスクに対して複数の生履歴レコードを生成していても、必要に応じて 1 件の Display Task に集約して扱う。

## 4. 対象範囲

- 対象内:
  - 実行履歴一覧を、タスク単位表示から run 単位表示へ再設計すること。
  - `run_id` をキーにした専用のクエリ実行結果詳細画面を提供すること。
  - 詳細画面内でタスク結果のドリルダウンを提供すること。
  - 詳細画面内で DAG を表示すること。
- 対象外:
  - オーケストレーションロジックそのものの変更。
  - エージェントのプロンプト戦略の変更。
  - 別途要件化されない限り、元のユーザークエリ本文を保存または表示すること。

## 5. 機能要件

### 5.1 run 単位の実行履歴一覧

- 実行履歴一覧は、Query Run ごとに 1 件のトップレベル項目を表示しなければならない。
- 実行履歴一覧は、生のタスク単位レコードをトップレベル項目として表示してはならない。
- 各履歴項目は、少なくとも以下を含まなければならない。
  - `run_id`
  - 実行日時
  - run ステータス
  - クエリ最終結果のプレビュー
- 実行履歴一覧には、ロール表示や失敗マーカーなどの補助情報を含めてもよいが、主たる内容は個別タスクではなく Query Run を表現しなければならない。
- 履歴項目は新しい実行日時順に並ばなければならない。
- 履歴項目をクリックすると、その `run_id` に対応する専用のクエリ実行結果詳細画面へ遷移しなければならない。

### 5.2 実行履歴一覧における run ステータス表現

- システムは、少なくとも以下の run 単位ステータスを扱えなければならない。
  - queued
  - running
  - completed
  - failed
- Query Run の実行中で最終結果がまだ存在しない場合、履歴項目は現在のステータスと、利用可能な範囲で最善の結果プレビューまたはステータスメッセージを表示しなければならない。
- Query Run が失敗した場合、履歴項目は失敗状態と、最新で意味のあるエラー要約を表示しなければならない。

### 5.3 クエリ実行結果詳細画面

- システムは、選択した Query Run に対する専用の詳細画面を提供しなければならない。
- 詳細画面は `run_id` で一意に参照可能でなければならない。
- 詳細画面は、選択した Query Run について以下のセクションを表示しなければならない。
  - クエリ最終実行結果
  - 実行タスク一覧
  - DAG
- 指定された `run_id` が存在しない場合、画面は not-found または empty state を表示しなければならない。
- Query Run がまだ実行中の場合、詳細画面は最新の取得可能データを表示し、実行中であることを明確に示さなければならない。

### 5.4 クエリ最終実行結果セクション

- 詳細画面は、選択した Query Run の最終実行結果を表示しなければならない。
- 最終実行結果セクションは、全タスクの生レコード一覧ではなく、ユーザー向けの正規の最終結果を表示しなければならない。
- 結果ペイロードが構造化 JSON の場合、UI はユーザー向け詳細画面として読みやすい形で表示しなければならない。
- 最終結果が生成される前に run が失敗した場合、最終結果セクションは失敗状態と最新の関連エラーペイロードを表示しなければならない。
- run が進行中の場合、最終実行結果が利用可能になるまでは、最終結果セクションは pending または running 状態を表示しなければならない。

### 5.5 実行タスク一覧セクション

- 詳細画面は、選択した Query Run で実行されたすべての Display Task 一覧を表示しなければならない。
- タスク一覧は、Planner の DAG `task_id` を、ユーザー向けの主要なタスク単位として使用しなければならない。
- タスク一覧は、Planner のトポロジ保存のためだけに存在する synthetic なメタデータ行を、実行タスクとして表示してはならない。
- バックエンドが 1 つの DAG タスクに対して複数の生レコードを保存している場合、UI はそれらを同一 Display Task の配下に集約し、別々のトップレベルタスクとして表示してはならない。
- この集約ルールは、内部ラウンドごとに別レコードが保存されうる Debate タスクで特に必須である。
- タスク一覧の順序は、DAG トポロジが利用可能な場合はそれに従うべきである。DAG トポロジが利用できない場合は、実行日時順をフォールバック順序としてよい。

### 5.6 タスク詳細の展開・折りたたみ動作

- 実行タスク一覧内の各タスク項目は、展開・折りたたみ可能でなければならない。
- 画面初期表示時は、全タスクの詳細領域が閉じていなければならない。
- タスク項目をクリックすると、そのタスクの詳細領域が展開または折りたたみされなければならない。
- 1 つのタスクを展開するために、全タスクを同時に展開することを要求してはならない。

### 5.7 タスク詳細の表示内容

- 各タスクの展開後詳細には、少なくとも以下を表示しなければならない。
  - `task_id`
  - タスク種別
  - 割り当てロール、または Debate タスクにおける参加者要約
  - 親タスク ID 一覧
  - 利用可能な場合、Planner が割り当てた動的パラメータ
  - 実行ステータス
  - 実行結果ペイロード
- 実行時パラメータが取得可能な場合、タスク詳細はそれらも表示すべきである。例として、モデル名、推論パラメータ、実行時に記録されたメタデータがある。
- Debate タスクについては、トップレベルのタスク詳細には、まず Debate タスク全体の集約結果を主結果として表示しなければならない。ラウンド単位の内部レコードは、同じ展開領域内で表示してもよいが、別のトップレベルタスク項目へ昇格させてはならない。

### 5.8 DAG セクション

- 詳細画面は、選択した Query Run の DAG を表示しなければならない。
- DAG は、同じ `run_id` に対して保存された Planner トポロジと一致していなければならない。
- DAG 表示は、少なくとも以下のタスク実行ステータスを視覚的に表現しなければならない。
  - pending
  - running
  - completed
  - failed
- DAG セクションは、実行タスク一覧と同じタスク集合を表現しなければならず、ユーザーが同一 run について構造と詳細出力の両方を理解できるようにしなければならない。

## 6. データ要件

- システムは、`run_id` をキーとした run 単位のデータ取得をサポートしなければならない。
- システムは、`run_id` をキーとした詳細取得をサポートし、少なくとも以下を返せなければならない。
  - クエリ最終実行結果
  - run ステータス
  - 当該 run のタスク集合
  - 当該 run の DAG トポロジ
- run 単位の集約は決定的でなければならず、UI 側のタイミング前提に依存してはならない。
- 主要なグルーピングキーは、Query Run に対しては `run_id`、Display Task に対しては Planner の `task_id` としなければならない。
- 生の履歴レコードが内部実行ステップ用の複合 task ID を使っている場合、詳細画面へ表示する前に、それらを対応する Display Task へ正しくマッピングしなければならない。

## 7. UI / UX 要件

- すべてのユーザー向けラベルとメッセージは日本語で表示しなければならない。
- 実行履歴一覧は、タスク詳細確認ではなく、クエリ結果の素早い一覧把握に最適化しなければならない。
- 詳細画面は、初期表示で情報過多にしないまま、必要時にドリルダウン確認できる構成でなければならない。
- 実行履歴一覧と詳細画面の双方について、loading、empty、error state を定義しなければならない。
- 詳細画面は、デスクトップだけでなく比較的小さいビューポートでも利用可能でなければならない。

## 8. 受け入れ基準

- 実行履歴一覧は、生のタスク実行レコード 1 件ごとの表示ではなくなっている。
- 実行履歴一覧は Query Run ごとに 1 件を表示し、各項目はクリック可能である。
- 履歴項目をクリックすると、同じ `run_id` の詳細画面が開く。
- 詳細画面には、その run の最終クエリ実行結果が表示される。
- 詳細画面には、その run の実行タスク一覧が表示される。
- 初期表示時、全タスク詳細は閉じている。
- ユーザーは任意の 1 タスクを展開し、そのパラメータと実行結果を確認できる。
- 詳細画面には、同じ run の DAG が表示される。
- Debate タスクの内部レコードは、タスク一覧で別のトップレベルタスクとして表示されない。
